# 剧情对话格式规范

玩家端任务详情页的「现场对话与处理方案」会读取 Ink 剧情文本，并按行解析为**旁白**或**角色对话气泡**。

推荐录入人员优先使用 **`[说话人] 文本`** 标准格式；旧式 **`「……」`** 引号对话仍兼容，但会在内容检查中提示迁移。

---

## 标准格式（推荐）

每行一条发言或旁白，行首写 `[说话人]`，后跟正文：

```text
[旁白] 启动会的咖啡还没凉，甲方代表把组织架构草稿拍在会议桌上。
[甲方代表] 总包、监理、设计、造价——接口人都在了，但有三块职责重叠。
[资料员] 名单今晚能发吗？各方都在等。
```

### 规则

| 规则 | 说明 |
|------|------|
| 一行一条 | 每条旁白或对话单独一行，不要混在同一行 |
| `[旁白]` | 渲染为灰色旁白块，不显示对话气泡 |
| `[角色名]` | 渲染为带角色名的蓝色对话气泡 |
| 角色名 | 使用中文岗位/身份，如 `甲方代表`、`资料员`、`监理` |
| 无标记行 | 无法识别时按**旁白**处理 |

### 示例（第一章样板）

参考 Ink 源数据（生成后位于 `src/ink/stories/`）：

- `src/data/stageInkStories.ts` → `setup_project_team.ink`
- `src/data/stageInkStories.ts` → `coordinate_first_meeting.ink`

修改剧情正文请编辑 `stageInkStories.ts` 中的 `intro` 字段，然后运行 `npm run ink:compile`。

---

## 旧格式（兼容，不推荐新增）

仍支持在同一行内混用旁白与 `「引号对话」`：

```text
甲方代表敲敲杯子：「今天别又开成讨论会——散会时我要能签字的纪要。」
```

旧格式会通过关键词猜测说话人，**不稳定**。新录入内容请改用 `[说话人]` 格式。

---

## Ink 文件注意事项

1. **不要改 Ink 语法**：选项、跳转、`-> END` 等保持原样，只改纯文本段落。
2. **选项与结局段落**：处理方案、结局描述可继续用普通旁白，不必加 `[说话人]`。
3. **修改后编译**：编辑 `src/data/stageInkStories.ts` 或对应 `.ink` 后运行 `npm run ink:compile`（或 `npm run build`）生成 JSON。
4. **内容检查**：`npm run content:check` 会对仍含 `「……」` 且无 `[说话人]` 的 Ink 文件给出 **warning**，不会阻断构建。

## 常用说话人参考

| 说话人 | 适用场景 |
|--------|----------|
| 旁白 | 场景描述、动作、心理、结果叙述 |
| 甲方代表 | 甲方、业主、领导口径 |
| 资料员 | 资料、台账、名单 |
| 监理 | 监理、整改、签字 |
| 总包 | 总包、班组、现场负责人 |
| 设计院 | 设计、图纸、方案 |
| 造价咨询 | 造价、控制价、合同边界 |
| 物业 | 物业、钥匙、移交 |
| 消防专家 | 消防、通道、泵房 |

可按剧情需要扩展，前端会按 `[ ]` 内文字原样显示。

## 录入检查清单

- [ ] 对话行使用 `[说话人] 文本`
- [ ] 场景描写使用 `[旁白] …` 或普通无标记行
- [ ] 未新增仅依赖 `「……」` 的旧式对话
- [ ] 已运行 `npm run ink:compile`
- [ ] `npm run content:check` 无新增 fail
